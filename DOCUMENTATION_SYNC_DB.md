# Документация по синхронизации БД и управлению триггерами

Данный документ описывает процесс синхронизации данных между базами данных `one` (источник) и `two` (приемник), а также управление триггерами для отслеживания изменений.

## 1. Архитектура синхронизации

Синхронизация реализована по принципу **Transactional Outbox** с использованием механизмов PostgreSQL:
1. **Триггеры** на таблицах в базе `one` записывают изменения в таблицу `pg_events`.
2. После записи в `pg_events` вызывается `pg_notify`, уведомляющий слушателей о новом событии.
3. Команда `pg-events:apply` слушает уведомления и применяет изменения к базе `two`.

---

## 2. Первичная синхронизация (Full Sync)

Для полной синхронизации структуры и данных из `one` в `two` используются следующие команды:

### Синхронизация конкретной схемы
```bash
php artisan db:sync {schema_name}
```
**Что делает:**
- Удаляет существующие таблицы и последовательности в целевой схеме на базе `two`.
- Копирует структуру таблиц.
- Копирует данные пачками (chunks).
- Синхронизирует значения последовательностей (sequences).

### Синхронизация всех схем
```bash
php artisan db:sync_all
```
Выполняет `db:sync` для всех схем, найденных в базе `one`.

---

## 3. Управление триггерами

Триггеры необходимы для автоматического наполнения таблицы `pg_events` при любых изменениях (INSERT, UPDATE, DELETE) в базе данных.

### Установка триггеров
Чтобы установить триггеры на все таблицы (кроме системных и технических):
```bash
php artisan pg-events:setup-triggers --connection=one
```
*Примечание: По умолчанию команда может использовать соединение `two`, поэтому всегда указывайте `--connection=one` для источника.*

**Что происходит при установке:**
1. Создается/обновляется хранимая функция `fn_pg_events_notify()`.
2. Для каждой таблицы создается триггер `trg_pg_events_{schema}_{table}`, который срабатывает `AFTER INSERT OR UPDATE OR DELETE`.

### Снятие триггеров
Если необходимо временно или постоянно отключить отслеживание изменений:
```bash
php artisan pg-events:setup-triggers --connection=one --remove
```

## 4. Инкрементальная синхронизация (Real-time)

После того как триггеры установлены, необходимо запустить процесс прослушивания событий.

### Запуск обработчика событий
```bash
php artisan pg-events:apply
```

**Основные параметры:**
- `--channel=pg_events`: Имя канала `pg_notify` (должно совпадать с тем, что прописано в триггерах).
- `--timeout=10`: Время ожидания новых событий в секундах.
- `--checkpoint=default`: Имя файла для сохранения ID последнего обработанного события (хранится в `storage/app/pg-events-checkpoint-...`).
- `--dry-run`: Показать, какие изменения будут применены, без реального выполнения SQL-запросов.

**Логика работы:**
1. Команда подключается к базе `one` и выполняет `LISTEN pg_events`.
2. При получении уведомления извлекает данные из таблицы `pg_events`.
3. Формирует и выполняет соответствующий SQL-запрос (Insert/Update/Delete) в базе `two`.
4. Обновляет файл контрольной точки (checkpoint), чтобы при перезапуске начать с того же места.

---

## 5. Структура таблицы pg_events

Таблица `pg_events` в базе `one` служит очередью событий:

| Поле | Описание |
| :--- | :--- |
| `id` | Инкрементальный ID события |
| `event_type` | Тип (например, `public.users.insert`) |
| `aggregate` | Имя таблицы |
| `aggregate_id` | Первичный ключ затронутой записи |
| `payload` | JSON с данными записи (для UPDATE содержит `old` и `new` состояния) |
| `created_at` | Время возникновения события |

---

## 6. Рекомендации по эксплуатации

1. **Порядок действий:**
   - Сначала выполните полную синхронизацию через `db:sync-all`.
   - Затем установите триггеры через `pg-events:setup-triggers`.
   - Запустите `pg-events:apply` для поддержания актуальности.
2. **Мониторинг:**
   - Следите за размером таблицы `pg_events`. Рекомендуется периодически очищать старые обработанные события.
   - Проверяйте логи команды `pg-events:apply` на наличие ошибок (например, конфликты внешних ключей в базе `two`).
3. **Отказоустойчивость:**
   - Команда `pg-events:apply` использует чекпоинты. Если процесс упадет, после перезапуска он продолжит работу с последнего успешно примененного события.
